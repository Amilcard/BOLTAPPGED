BUG 1 — Affichage "6-17 ANS" sur les carousels Critique

FRONT DATA FLOW

Symptôme observé

Sur la page d'accueil, ALPINE SKY CAMP et WEST COAST SURF CAMP affichent "6-17 ANS" au lieu de leurs vraies tranches d'âge.

Données Supabase vérifiées

Séjourgd_stays.age_mingd_stays.age_maxgd_stay_sessions
ALPINE SKY CAMP1217Aucune entrée (ou sans age_min/age_max)
WEST COAST SURF CAMP1517Aucune entrée (ou sans age_min/age_max)

Les âges sont corrects dans gd_stays (12-17 et 15-17), mais le front ne les utilise jamais.

Cause racine

Fichier : app/page.tsx (lignes 15-16, 41)

La page d'accueil charge les âges uniquement depuis gd_stay_sessions :

// app/page.tsx — ligne 15-16
const agesData = await supabaseGed.from('gd_stay_sessions')
  .select('stay_slug, age_min, age_max, start_date, end_date')

Puis à la ligne 41, getStayAgeData(sessions) est appelé avec ce tableau. Si un séjour n'a aucune entrée dans gd_stay_sessions, le tableau est vide.

Fichier : lib/age-utils.ts (lignes 152-159)

La fonction calculateGlobalAgeRange() retourne alors les valeurs par défaut hardcodées :

export function calculateGlobalAgeRange(
  sessions: SessionAgeData[],
  defaultMin: number = 6,   // ← Problème ici
  defaultMax: number = 17   // ← Problème ici
): { ageMin: number; ageMax: number } {
  if (!sessions || sessions.length === 0) {
    return { ageMin: defaultMin, ageMax: defaultMax }; // → 6-17 !
  }
  ...
}

Au lieu d'utiliser gd_stays.age_min/age_max (12/17 ou 15/17), le code retourne 6-17.

Correction proposée

1. Dans app/page.tsx — Passer age_min/age_max de gd_stays comme fallback :

// Ligne 41 — Avant
const { ageMin, ageMax, ageRangesDisplay } = getStayAgeData(sessions);

// Ligne 41 — Après
const { ageMin, ageMax, ageRangesDisplay } = getStayAgeData(
  sessions,
  sejour.age_min ?? 6,  // fallback depuis gd_stays
  sejour.age_max ?? 17  // fallback depuis gd_stays
);

2. Dans lib/age-utils.ts — Accepter les fallbacks dans getStayAgeData() :

export function getStayAgeData(
  sessions: SessionAgeData[],
  fallbackMin: number = 6,
  fallbackMax: number = 17
) {
  const { ageMin, ageMax } = calculateGlobalAgeRange(sessions, fallbackMin, fallbackMax);
  ...
}

3. Même correction dans app/sejour/[id]/page.tsx (ligne 37) :

const { ageMin, ageMax, ageRangesDisplay } = getStayAgeData(
  sessionAgeData,
  stay.age_min ?? 6,
  stay.age_max ?? 17
);

Impact : Tous les séjours sans entrée dans gd_stay_sessions afficheront désormais les âges de gd_stays au lieu de "6-17 ANS".

BUG 3 — WEST COAST : Sync n8n échouée Majeur

N8N DATA

Symptôme observé

Session 05/07 → 16/07 marquée "complet" sur UFOVAL, mais is_full = false dans Supabase.

Données Supabase vérifiées

gd_session_prices pour surf-sur-le-bassin, session 05/07→16/07 : is_full = false

Contrairement à ALPOO KIDS, ici la donnée BDD est fausse. Le workflow n8n n'a pas synchronisé.

Cause probable

Le workflow n8n (01_UFOVAL_IS_FULL_SESSIONS_v4.json) fait un PATCH sur gd_session_prices en matchant sur stay_slug + start_date + end_date. Si les dates scrapées sur UFOVAL diffèrent d'un jour par rapport à celles en BDD (convention inclusive vs exclusive), le match échoue silencieusement.

Actions à mener

  • Correction immédiate : UPDATE manuel dans Supabase :
UPDATE gd_session_prices
SET is_full = true, updated_at = NOW()
WHERE stay_slug = 'surf-sur-le-bassin'
  AND start_date = '2026-07-05'
  AND end_date = '2026-07-16';
  • Correction pérenne : Vérifier les logs n8n pour surf-sur-le-bassin, ajuster le matching de dates si nécessaire (tolerance ±1 jour)
