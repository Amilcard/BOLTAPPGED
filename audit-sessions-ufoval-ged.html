Cause probable : Les dates ont été importées initialement dans gd_stay_sessions et gd_session_prices lors du scraping UFOVAL. Le workflow n8n (01_UFOVAL_IS_FULL_SESSIONS_v4.json) ne met à jour que le champ is_full — il ne corrige jamais les dates. Si UFOVAL a modifié ses dates de sessions après l'import initial, ou si le parsing des dates françaises a eu un décalage (ex : "samedi 25 juil." interprété comme le 26), la BDD GED reste désynchronisée.
Correctifs recommandés :
Court terme : Corriger manuellement les end_date (et start_date pour Alpine Sky Camp août) directement en BDD Supabase dans les tables gd_session_prices ET gd_stay_sessions.
Moyen terme : Enrichir le workflow n8n pour qu'il compare et mette à jour les dates scrappées (pas seulement is_full), avec un flag date_changed pour traçabilité.

2. Complétudes non synchronisées (n8n) 2 anomalies

Sessions marquées "Complet" sur UFOVAL mais encore affichées comme disponibles sur GED (ou inversement). Le workflow n8n fait un PATCH gd_session_prices.is_full toutes les 6h, mais le matching repose sur stay_slug + start_date + end_date.

Séjour Session Statut UFOVAL Statut GED Cause probable
Alpoo Kids 5 → 11/07 COMPLET Disponible Le PATCH n8n matche sur start_date + end_date — si les dates GED diffèrent, le PATCH ne trouve pas la row et le is_full reste false
West Coast Surf Camp 5 → 16/07 COMPLET Disponible Même problème de matching dates. Si les dates GED ne correspondent pas exactement aux dates scrappées, la requête eq.start_date.eq.end_date ne matche rien.
Analyse technique : Le workflow n8n v4 fait un PATCH avec ce pattern :
gd_session_prices?stay_slug=eq.{slug}&start_date=eq.{scraped_start}&end_date=eq.{scraped_end}
Si les dates en BDD sont décalées par rapport aux dates scrappées sur UFOVAL (cf. section 1), le PATCH retourne 0 rows updated et le statut is_full n'est jamais mis à jour. C'est un effet domino : les erreurs de dates provoquent les erreurs de complétude.
Correctifs recommandés :
1. Corriger d'abord les dates (section 1) → le prochain cycle n8n mettra naturellement à jour les is_full.
2. Ajouter dans le workflow n8n un log/alerte quand un PATCH retourne 0 rows (= session non trouvée en BDD).
3. Envisager un matching plus souple : stay_slug + start_date uniquement (sans end_date) si une seule session commence à cette date.

3. Tranches d'âge carrousel incohérentes 2 anomalies

Les tranches d'âge affichées sur les cartes carrousel (StayCard) ne correspondent pas au public cible réel du séjour. L'affichage utilise stay.ageRangesDisplay calculé depuis gd_stay_sessions.age_min / age_max, avec un fallback sur gd_stays.age_min / age_max (défaut : 6–17).

Séjour Âge affiché (carrousel) Âge réel (cible) Source de l'erreur
Alpine Sky Camp 6-17 ANS 15-17 ANS gd_stay_sessions.age_min/age_max contient probablement les valeurs par défaut (6/17) au lieu des valeurs réelles (15/17), OU aucune session n'a de données d'âge → fallback sur gd_stays.age_min=6, age_max=17 (défaut schema)
West Coast Surf Camp 6-17 ANS 12-17 ANS Même problème : les champs age_min/age_max dans gd_stay_sessions ou gd_stays utilisent les valeurs par défaut au lieu des valeurs spécifiques au séjour
Analyse technique : Le code dans page.tsx (home) calcule les âges via getStayAgeData(sessions) qui fait un Math.min/Math.max sur toutes les sessions. Si les sessions ont age_min=0, age_max=0 (ou null → converti en 0), le calcul retourne 0-0 et le fallback gd_stays.age_min=6, age_max=17 s'applique. Le StayCard affiche alors "6-17 ANS" par défaut, quel que soit le public réel.
Correctifs recommandés :
Court terme : Mettre à jour gd_stays.age_min et gd_stays.age_max pour Alpine Sky Camp (15/17) et West Coast Surf Camp (12/17) directement en BDD.
Court terme : Mettre à jour gd_stay_sessions.age_min / age_max pour chaque session de ces séjours avec les bonnes valeurs.
Moyen terme : Ajouter une validation dans le script d'import qui alerte quand age_min/age_max = valeurs par défaut (6/17) pour des séjours clairement ciblés ados.
